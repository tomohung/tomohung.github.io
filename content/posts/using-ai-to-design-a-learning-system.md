拿到一份交易學習課程，第一步就是把所有影音內容轉成筆記。

但轉完之後，面對一大批資料，才發現真正的挑戰才剛開始——**量太大，不知道從哪裡讀起，也不知道怎麼讀才有效。**

---

## 從整理筆記，到設計學習系統

今天試著把這批筆記丟給 AI 討論，想搞清楚一件事：**面對大量課程資料，怎麼建立一套真正有效的學習方式？**

討論過程中，逐漸整理出一個思路。光是「讀完」不夠，需要的是一套系統：有順序、有節奏、有方法檢核自己到底學進去沒有。

最後請 AI 把這個流程輸出成一個可重複使用的 prompt，以後不管拿到什麼課程，都可以套用同樣的設計邏輯。

---

## 學習系統的五個步驟

這套系統的核心是請 AI 扮演「學習系統設計師」，依序完成五件事：

**第一步，掃描與分類。** 把所有資料讀過一遍，找出核心概念框架，就像先看清楚一棵樹的主幹，再去看枝葉。

**第二步，建立學習地圖。** 把資料切成每個約 20 分鐘的學習單元，依邏輯排序。每個單元有兩組問題：**預習問題**讓你帶著目的去讀，不是被動接收，而是主動找答案；**檢核問題**讓你讀完後確認自己真的理解了，而不是「感覺讀懂了」。

**第三步，建立學習進度表。** 每個單元對應一列，用簡單的符號標記狀態，清楚看到自己在哪裡。

**第四步，建立複習排程。** 採用間隔重複的邏輯——讀完後第 1、3、7 天各複習一次，在記憶還沒完全消退前及時強化。

**第五步，建立實戰模板。** 如果課程有實作性質（像交易課程），就額外建立一份推演模板，把學到的框架轉成可執行的分析步驟。

---

## 背後的學習科學

這套設計不是直覺，是有依據的。

| 機制 | 原理 | 對應到系統的哪裡 |
|------|------|------|
| 主動回憶 | 主動提取記憶比被動複習更有效 | 預習問題、檢核問題 |
| 間隔重複 | 在遺忘曲線的關鍵點複習，強化長期記憶 | 複習排程（+1 / +3 / +7 天）|
| 精緻化 | 將新知識與已有知識連結 | 預習問題「從已知到未知」的設計 |
| 交錯練習 | 混合不同主題複習，提升遷移能力 | 每週總複習 |
| 刻意練習 | 在真實情境中應用所學 | 實戰推演模板 |

## Prompt 模板：可直接複製使用

整個流程最後被整理成一個可重複使用的 prompt。把 `{{}}` 的部分替換成你的實際資料，就能讓 AI 輸出一整套量身訂製的學習計畫。

```
你是一位學習系統設計師。我有一批課程學習資料，請幫我建立一套完整的結構化學習系統。

## 資料描述

- 課程主題：{{課程主題，例如：台指期當沖交易}}
- 資料位置：{{資料夾路徑或描述}}
- 資料格式：{{例如：Obsidian markdown 筆記 / PDF / 影片逐字稿 / 純文字}}
- 資料數量：{{大約幾篇/幾個檔案}}
- 資料來源：{{例如：某講師的 YouTube 影片筆記、線上課程講義}}

## 我的背景

- 對這個主題的基礎程度：{{從零開始 / 略懂 / 有基礎 / 進階}}
- 每天可投入的學習時間：{{例如：20-30 分鐘}}
- 學習目標：{{你希望達到什麼程度？能做到什麼？}}

## 請你完成以下步驟

### 第一步：掃描與分類

1. 掃描所有資料檔案，讀取標題、標籤、摘要等資訊
2. 識別資料的組織邏輯（章節、主題、難度等）
3. 找出核心概念框架（類似知識樹的主幹）

### 第二步：建立學習地圖

根據掃描結果，建立 `學習地圖.md`：

1. 將所有資料分組為學習單元，每單元控制在約 20 分鐘閱讀量
2. 依邏輯順序排列（基礎→進階，或依主題框架）
3. 每個單元包含：
   - **預習問題**（3-5 題）：從已知連結到未知，引導主動閱讀
   - **閱讀後檢核問題**（3-5 題）：概念理解、情境應用、跨單元連結

預習問題設計原則：
- 喚起已有知識或經驗
- 引導注意該單元的重點
- 讓閱讀變成「找答案」而非被動接收

檢核問題設計原則：
- 能用自己的話解釋核心概念
- 能判斷情境中該如何應用
- 能與前面學過的內容建立連結

### 第三步：建立學習進度表

建立 `學習進度.md`：
- 對應學習地圖的每個單元
- 欄位：單元 / 主題 / 預習 / 閱讀 / 複習1 / 複習2 / 複習3 / 備註
- 使用 ⬜ / ✅ 標記狀態

### 第四步：建立複習排程

建立 `複習排程.md`：
- 採用間隔重複：完成閱讀後第 1、3、7 天各複習一次
- 提供完整的複習日曆，標示每天需要複習哪些單元
- 包含每週總複習的安排與方法

### 第五步：建立實戰模板（如適用）

如果這個課程有實作/應用的性質，建立 `實戰推演模板.md`：
- 按照課程的核心框架設計分析步驟
- 每個步驟提供 checklist
- 包含執行紀錄和覆盤檢討區

## 輸出要求

- 所有檔案使用繁體中文
- 使用 Obsidian 相容的 markdown 格式（wiki link `[[]]` 語法）
- 英文專有名詞保留原文
- 如果同一主題有基礎版和進階版的資料，在學習地圖中標註並安排交叉參照
```

## 為什麼要把流程變成 prompt？

因為**這個問題不是只有這次會遇到。**

只要拿到任何一批課程資料——不管是投資、程式、語言、設計——面對的挑戰都一樣：資料太多、不知道怎麼規畫、讀完沒有留下來。

把設計邏輯沉澱成一個可重複使用的工具，下次就不用從頭想，直接套用，把精力放在學習本身。
